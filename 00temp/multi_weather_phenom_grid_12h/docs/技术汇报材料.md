# 网格天气现象电码生成系统 — 技术汇报材料

**系统名称**：网格天气现象综合电码生成系统
**标准依据**：QX/T 740-2024《基于网格预报的城镇预报生成规范 天气现象》
**架构规范**：国省统筹算法技术规范（三级架构）
**版本**：v1.0.0

---

## 一、项目背景与目标

- **业务背景**：城镇天气预报需要基于网格数值预报产品，自动生成标准化的天气现象综合电码，替代人工判识，提升预报生产效率与规范性。
- **核心目标**：读取网格预报 NC 数据（R03/PTYPE03/TCC/FOG/HAZE/SAND/THUNDER/HAIL 等9个变量），依据国家行业标准自动判识 **31种天气现象**，并输出标准的 **5位综合电码网格**（NetCDF4，meteva 6D格式）。
- **覆盖时效**：起报后 003~240 小时，按 12h 为一个时段，共 **20个时段**。

## 二、系统架构

### 2.1 三级架构 + cli/src 分层

```
┌─────────────────────────────────────────────────────────┐
│  Level 3  算法应用 (CLI)                                  │
│           cli/main.py     — 命令行入口，POSIX 退出码       │
│           cli/runner.py   — 编排调度：加载 + 计算，并行     │
│           cli/data_loader.py — 唯一的文件I/O入口(meteva)   │
├─────────────────────────────────────────────────────────┤
│  Level 2  算法插件 (Plugins)                              │
│           DIA_WeatherPhenomIdentifier   判识               │
│           DIA_WeatherPhenomSelector     选取               │
│           DIA_WeatherPhenomLogicJudger  逻辑关系判断        │
│           DIA_WeatherPhenomEncoder      编码               │
├─────────────────────────────────────────────────────────┤
│  Level 1  基础函数 (Functions)                            │
│           identify / select / judge / encode / decode    │
└─────────────────────────────────────────────────────────┘
```

**分层职责边界**（cli 与 src 严格解耦，是本项目的核心架构决策）：

| 层级 | 模块 | 职责 | 文件I/O |
|------|------|------|---------|
| cli | `data_loader.py` | 读取NC文件，产出内存数据 | **是**（唯一入口） |
| cli | `runner.py` | 编排：加载(cli.data_loader) → 计算(src.processor) | 否 |
| cli | `main.py` | 命令行参数解析 | 否 |
| src | `processor.py` 等算法模块 | 纯内存计算（判识/选取/逻辑/编码）+ 保存输出 | 否（只接收内存数据） |

> src 层不 `import meteva`、不感知任何数据加载细节，保证算法层**环境无关、可独立单测**（mock 即可，无需真实NC文件）。

### 2.2 配置模块职责分离

```
resource/
  ├── data_schema.py     ← 算法结构常量（环境无关，如变量分组/时效上限）
  ├── config.py          ← 运行环境配置（数据根目录，部署相关）
  └── weather_config.py  ← 天气现象业务规则（电码/阈值/互斥关系，环境无关）
```

## 三、核心算法流程

依据 QX/T 740-2024 附录C 流程图，5步流程：

```
输入变量(NC) → [cli] 数据加载 → [src] 判识 → [src] 选取 → [src] 逻辑关系判断 → [src] 编码 → 输出NC
```

| 步骤 | 模块 | 说明 |
|------|------|------|
| Step1 数据加载 | `cli.data_loader.load_segment()` | 读取9个变量，返回 `{变量名: ndarray[4,lat,lon]}` |
| Step2 判识 | `DIA_WeatherPhenomIdentifier` | 依附录A判识31种天气现象是否出现 |
| Step3 选取 | `DIA_WeatherPhenomSelector` | 选取影响级最高的A现象 + 独立出现的B现象（int8索引） |
| Step4 逻辑判断 | `DIA_WeatherPhenomLogicJudger` | 判断A/B关系：单一(0)/转(1)/间(2)/伴有(3) |
| Step5 编码 | `DIA_WeatherPhenomEncoder` | 生成5位电码：`K*10000 + AA*100 + BB` |

**综合电码格式**（示例）：`10207` = 阴转小雨，`20001` = 晴间多云，`35661` = 中度霾伴有轻雾

## 四、关键设计与性能优化

| 设计点 | 内容 |
|--------|------|
| 全 int8 索引运算 | selector/logic_judger 内部以 `int8` 索引代替字符串，仅输出阶段查表转电码，规避 `dtype=object` 性能开销 |
| 预计算矩阵 | 模块加载时一次性预计算互斥矩阵、伴有矩阵、间候选矩阵、电码查找表(LUT) |
| Plugin 单例 | 4个 Plugin 在 `processor.py` 中以模块级单例持有，避免重复初始化 |
| TCC 单位自适应 | 自动识别 TCC 为 0~1 或 0~100，`max≤1.1` 时自动 ×100 |
| 并行策略 | `cli/runner.py`：时段间线程池并行(`max_seg_workers`) + 单时段内文件级并行读取(`max_workers`)，均由cli层负责，src层无感知 |

## 六、数据规格

### 6.1 输入变量

| 变量名 | 说明 | 有效时效 |
|--------|------|----------|
| R03 | 3h累计降水量(mm) | 003~240h |
| PTYPE03 | 降水类型(雨/雪/雨夹雪/冻雨) | 003~240h |
| TCC | 总云量(%或0~1) | 003~240h |
| FOG / HAZE / SAND | 雾/霾/沙尘等级 | 003~072h |
| THUNDER / HAIL | 雷暴/冰雹标志 | 003~072h |

> 72h后缺失的短时效变量自动填零，不影响降水类和天空类判识。

### 6.2 输出格式

```
{output_dir}/{YYYYMMDDHH}/{YYYYMMDDHH}.{TTT}.nc
```

采用 **meteva 标准 6D DataArray** 格式，维度 `[member, level, time, dtime, lat, lon]`，变量名 `phenom_code`，可直接用 `meteva.base.read_griddata_from_nc()` 读取。

## 七、项目成果总结

1. **架构层面**：完成 cli（I/O+编排）与 src（纯内存算法）的彻底解耦，符合国省统筹算法技术规范的三级架构与Plugin接口要求
2. **算法层面**：实现31种天气现象、4类逻辑关系（单一/转/间/伴有）的完整判识与5位电码编码，全链路 int8 索引优化性能
3. **质量保障**：58个单元测试覆盖5个核心模块的正常路径、边界条件与异常降级场景，全部通过；并完成真实生产数据端到端验证
4. **可维护性**：配置（数据结构常量/运行环境/业务规则）三层分离，部署环境变更无需改代码
